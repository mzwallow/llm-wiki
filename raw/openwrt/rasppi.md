The Raspberry Pi is a series of [single-board computer](https://en.wikipedia.org/wiki/single-board%20computer "https://en.wikipedia.org/wiki/single-board computer") s using Broadcom SoCs and lack an [Ethernet Network Switch](https://openwrt.org/docs/techref/hardware/switch "docs:techref:hardware:switch").

Raspberry Pi 1, 2, and 3 are compiled with the ARM hard-float kernel ABI, this means that non-integer math is done in hardware instead of in software. Those series have somewhat reached legacy status.

Raspberry Pi 4 has significantly higher spec CPU, RAM, and storage while maintaining the form factor, low power consumption, and passive cooling. Its capability can exceed many top tier routers. However it requires adding a USB 3.0 to gigabit ethernet adapter, gigabit switch, and wireless access point to reach similar networking functionality.

Raspberry Pi 5 released in late 2023 with improvements including a power button, RTC, and 2-3x the performance of the Pi 4.

[![](https://openwrt.org/_media/media/raspberry_pi_foundation/raspberry-pi-4-model-b.jpg?w=350&tok=339aea)](https://openwrt.org/_detail/media/raspberry_pi_foundation/raspberry-pi-4-model-b.jpg?id=toh%3Araspberry_pi_foundation%3Araspberry_pi "media:raspberry_pi_foundation:raspberry-pi-4-model-b.jpg")

The Raspberry Pi is supported in the **bcm27xx** target.

Subtargets are **bcm2708** for Raspberry Pi 1, **bcm2709** for the Raspberry Pi 2, **bcm2710** for the Raspberry Pi 3, and **bcm2711** for the Raspberry Pi 4. Most subtargets feature 32-bit and 64-bit builds in line with the official Raspberry Pi OS which is also Linux based.

1. Download the appropriate **bcm27xx** image for your Raspberry Pi and desired OpenWrt branch from [Firmware Selector](https://firmware-selector.openwrt.org/?version=SNAPSHOT&target=bcm27xx%2Fbcm2711&id=rpi-4 "https://firmware-selector.openwrt.org/?version=SNAPSHOT&target=bcm27xx%2Fbcm2711&id=rpi-4").
2. Flash the image to a micro SD card using a disk imager, such as the open source one from the [Raspberry Pi team](https://www.raspberrypi.org/blog/raspberry-pi-imager-imaging-utility/ "https://www.raspberrypi.org/blog/raspberry-pi-imager-imaging-utility/").
3. Once complete, insert the SD card into your Raspberry Pi and power up. OpenWrt will boot.

#### Alternatively, flashing from the Linux command line

Plug in the SD card and check kernel messages with `dmesg`.

The most recent message should give you the SD card device name, **sdb** or **sdf** or similar. Ensure the drive was not auto-mounted by your OS.

From root, use **dd** to copy the image file to the device you identified, for example with Pi 4:

```js
gzip -d imagename-factory.img.gz
dd if=imagename-factory.img of=/dev/sdX bs=2M conv=fsync
sync
```

Replacing the **sdX** with your device name, such as **sdb** or **sdf**.

After booting, you can [expand the filesystem on the SD card](https://openwrt.org/docs/guide-user/installation/installation_methods/sd_card#expanding_the_filesystem "docs:guide-user:installation:installation_methods:sd_card").

Due to licensing uncertainty, OpenWrt images do not include the firmware for the wireless chips used in the Raspberry Pi Zero 2 W. To have working wireless interface manual intervention is needed, check this [e9f9cd14cc71](https://git.openwrt.org/e9f9cd14cc71 "https://git.openwrt.org/e9f9cd14cc71") commit message for details.

The Raspberry Pi bootloader EEPROM has many improvements since launch for performance, bug fixes, and power draw/ thermal throttling issues. Thus it is recommended to update the EEPROM. The [Raspberry Pi Imager](https://www.raspberrypi.com/software/ "https://www.raspberrypi.com/software/") allows to flash a minimal operating system on a usb stick or micro SD card which updates the EEPROM and can also change the boot priority. The OpenWrt package **bcm27xx-eeprom** provides rpi-eeprom-\* commands as provided by the Pi foundation.

The EEPROAM can also be updated with Raspberry Pi OS. See below for links to that effect:

1. [EEPROM github](https://github.com/raspberrypi/rpi-eeprom/tree/master "https://github.com/raspberrypi/rpi-eeprom/tree/master")
2. [Updating the Raspberry Pi 4/5 boot EEPROM](https://www.raspberrypi.org/documentation/computers/raspberry-pi.html#rpi-eeprom-update "https://www.raspberrypi.org/documentation/computers/raspberry-pi.html#rpi-eeprom-update")

The Pi 4 B originally shipped at 1.5GHz, later revisions include an improved onboard power supply and [ship at 1.8GHz](https://www.raspberrypi.com/news/bullseye-bonus-1-8ghz-raspberry-pi-4/ "https://www.raspberrypi.com/news/bullseye-bonus-1-8ghz-raspberry-pi-4/"). The Pi 5 ships at 2.4GHz. Users should be sure to attach the self-adhesive heatsinks.

Newer kernel versions in OpenWrt trunk feature a capability called “device tree support” which allows for automatic configuration of devices in /dev.

Putting the following lines at the end of `/boot/config.txt` (on the FAT32 partition) allows this feature to work:

```js
dtparam=i2c1=on (or dtparam=i2c0=on on old models)
dtparam=spi=on
dtparam=i2s=on
```

For I2C, it is also required to install the kernel module [kmod-i2c-bcm2835](https://openwrt.org/packages/pkgdata/kmod-i2c-bcm2835 "packages:pkgdata:kmod-i2c-bcm2835") that contains the Broadcom I2C master controller driver:

```js
opkg update
opkg install kmod-i2c-bcm2835
```

It might be tricky to connect via Ethernet for the first time. By default the DHCP client is disabled and the IP address is configured as static 192.168.1.1
After flashing OpenWrt as described above, you can reach the OpenWrt via Ethernet by using the IP address 192.168.1.1
It is highly likely that the gateway (router) to which the Raspberry Pi is connected has the same IP of 192.168.1.1
In that case if possible change the gateway IP to something else, then try accessing using the default static IP address of 192.168.1.1 using SSH or HTTP. Once you have access to the OpenWrt you can set password and enable DHCP.

If it is not possible to resolve the IP conflict read further...

In order to set it up as a DHCP client, attach a serial console and do the following:

```js
uci set network.lan.proto=dhcp
uci commit
/etc/init.d/network restart
```

Alternatively, in order to set it up with a fixed IP different from the default 192.168.1.1 do the following:

```js
uci set network.lan.ipaddr=192.168.1.199
uci commit
/etc/init.d/network restart
```

Now you should be able to log in via SSH or HTTP to set your password.

Alternativly, in order to get online with a keyboard and monitor directly on the PI, set the single port to 'wan' edit the network file, create the wan and change the lan to eth1

For example, to configure onboard ethernet to eth0 for LAN and USB adapter eth1 for WAN: `vi /etc/config/network`

```js
config device option name 'br-lan'
option type 'bridge'
list ports 'eth0'

config interface 'wan'
option device 'eth1'
option proto 'dhcp'

config interface 'wan6'
option device 'eth1'
option proto 'dhcpv6'
```

Reboot.

Be sure to change these back once your usb adapter is installed.

#### A note on VLANs

While most of sources advocate for using DSA and VLAN filtering on \`br-lan\` bridge, if your use case is wired only router - “on stick” or with extra Ethernet dongle/Pi Hat, consider not using bridge and use driver VLAN capacities. For example below is complete \`/etc/config/network\` that creates untagged LAN and tagged Guest and IoT networks. It is viable alternative to DSA complexity and the simplest configuration to work with popular enterprise grade APs / managed switches.

```js
config interface 'loopback'
        option device 'lo'
        option proto 'static'
        option ipaddr '127.0.0.1'
        option netmask '255.0.0.0'

config globals 'globals'
        option ula_prefix 'fd6d:abcd:ffff::/48'

config interface 'lan'
        option device   'eth0'
        option proto    'static'
        option ipaddr   '192.168.1.1'
        option netmask  '255.255.255.0'
        option ip6assign '60'

config interface 'wan'
        option device   'eth1'
        option proto    'dhcp'

config interface 'wan6'
        option device   'eth1'
        option proto    'dhcpv6'

config interface 'guest'
        option device   'eth0.2'
        option proto    'static'
        option ipaddr   '192.168.2.1'
        option netmask  '255.255.255.0'
        option ip6assign '60'

config interface 'iot'
        option device   'eth0.3'
        option proto    'static'
        option ipaddr   '192.168.3.1'
        option netmask  '255.255.255.0'
        option ip6assign '60'
```

It is common to use a USB 3.0 to Gigabit Ethernet adapter to connect your WAN /Internet device (e.g. cable modem), thus freeing up the built-in Ethernet port for a switch and/or wireless access point. For the Pi Zero you can plug-in an adapter via [USB-OTG](https://en.wikipedia.org/wiki/USB_On-The-Go "https://en.wikipedia.org/wiki/USB_On-The-Go") cable. However, system does not support it by default.

Install the kernel module associated with your adapter. It is suggested to use Firmware Selector, click Customize, and add your adapter kmod so it is ready use. Some common examples:

- RTL8153: kmod-usb-net-rtl8152
- AX8817: kmod-usb-net-asix-ax88179

Alternatively, you can compile your own **bcm27xx** image with these included, for example (on menuconfig):

```js
Kernel modules -> USB Support -> kmod-usb-net
kmod-usb-net-asix
asix-ax88179
```

Then proceed with the connecting instructions on the previous chapter. [Github guide](https://github.com/smeathers/lede-pi0w-usb/blob/master/README.md "https://github.com/smeathers/lede-pi0w-usb/blob/master/README.md").

Note: The Raspberry Pi 5 has its [own embedded RTC](https://www.raspberrypi.com/documentation/computers/raspberry-pi-5.html#real-time-clock-rtc "https://www.raspberrypi.com/documentation/computers/raspberry-pi-5.html#real-time-clock-rtc") and no overlay or configuration is needed.

The Raspberry Pi does not have a battery-backed hardware [Real-time Clock](https://en.wikipedia.org/wiki/Real-time%20Clock "https://en.wikipedia.org/wiki/Real-time Clock") (RTC). Instead, OpenWrt can be configured to be retrieving the current date and time from [network time-servers (NTP)](https://openwrt.org/docs/guide-user/advanced/ntp_configuration "docs:guide-user:advanced:ntp_configuration").

For usage scenarios where an internet connection is not guaranteed and correct time and date are important, adding an RTC may be desirable.

Below are instructions for installing an RTC that can connect to the Raspberry Pi I2C GPIO, such as the common ds1307 RTC:

First [enable the I2C](https://openwrt.org/toh/raspberry_pi_foundation/raspberry_pi#using_i2c_and_spi "toh:raspberry_pi_foundation:raspberry_pi").

Search for the appropriate kernel module and install it:

```js
opkg update
opkg list kmod-rtc-*
opkg install kmod-rtc-ds1307
```

Add the appropriate [Device Tree Overlay](https://github.com/raspberrypi/firmware/blob/master/boot/overlays "https://github.com/raspberrypi/firmware/blob/master/boot/overlays") line to the end of the `/boot/config.txt` file, in addition to `dtparam=i2c1=on`:

```js
dtoverlay=i2c-rtc,ds1307
```

Set the [timezone](https://openwrt.org/docs/guide-user/base-system/system_configuration "docs:guide-user:base-system:system_configuration") and [sync the date and time](https://openwrt.org/docs/guide-user/services/ntp/client-server "docs:guide-user:services:ntp:client-server"). This can be done by LuCI > System > System > Timezone. Then tick “Sync with NTP -Server” or “Sync with browser”.

For set the RTC time from the system time for the first time after inserting the battery to the RTC:

```js
hwclock -w
```

The `hwclock` tool should be included in [busybox](https://openwrt.org/packages/pkgdata/busybox "packages:pkgdata:busybox") which is already installed by default with OpenWrt. If not, install the package [hwclock](https://openwrt.org/packages/pkgdata/hwclock "packages:pkgdata:hwclock"):

```js
opkg update
opkg install hwclock
```

From now on, during the boot, OpenWrt will read and set the date according to the RTC. If the RTC is installed properly, a message about setting the time from the RTC will appear in the kernel messages log:

```js
dmesg | grep rtc
[    7.033548] rtc-ds1307 1-0068: registered as rtc0
[    7.039222] rtc-ds1307 1-0068: setting system clock to 2022-12-14T20:37:26 UTC (1671050246)
```

This can also be verified by disconnecting the Raspberry from the internet and reboot. If the RTC battery is OK, the date and time will be correct:

```js
date
```

#### Troubleshooting

If the RTC is not identified properly by `hwclock`, install the package [i2c-tools](https://openwrt.org/packages/pkgdata/i2c-tools "packages:pkgdata:i2c-tools"):

```js
opkg update
opkg install i2c-tools
i2cdetect -y 1
```

The output should include “UU”, which means that an I2C device has been detected and is being used by a driver.

An output of hyphens only “- -” means that no any I2C device has been detected. Check your RTC and the battery.

An output that include a number, means that a I2C device has been detected, but a driver does not use it. Check if the correct kernel module is installed and the correct dtoverlay line is added to the `/boot/config.txt` file.

If an error is received, such as:

```js
Error: Could not open file \`/dev/i2c-1' or \`/dev/i2c/1': No such file or directory
```

Make sure the appropriate i2c dtparam line is added to the `/boot/config.txt` file and the i2c controller driver is installed.

Note: The Raspberry Pi 5 has its own [PWM FAN connector](https://www.raspberrypi.com/documentation/computers/raspberry-pi-5.html#active-cooler "https://www.raspberrypi.com/documentation/computers/raspberry-pi-5.html#active-cooler") and no overlay or configuration is needed.

To install [the official case fan](https://www.raspberrypi.com/products/raspberry-pi-4-case-fan/ "https://www.raspberrypi.com/products/raspberry-pi-4-case-fan/") and other GPIO fans:

```js
opkg update
opkg install kmod-hwmon-gpiofan
```

Add the gpio-fan [Device Tree Overlay](https://github.com/raspberrypi/firmware/blob/master/boot/overlays "https://github.com/raspberrypi/firmware/blob/master/boot/overlays") line to the end of the `/boot/config.txt` file and configure it:(The example below is taken from the Pi OS configuration.)

```js
dtoverlay=gpio-fan,gpiopin=14,temp=80000
```

The gpio-fan dtoverlay is used to configure a GPIO pin to control a cooling fan.

The value of the parameter “gpiopin” represents the GPIO pin number.

The value of the parameter “temp” represents the temperature at which the fan switches on, in millicelsius. See [gpio-fan](https://github.com/raspberrypi/firmware/blob/master/boot/overlays/README "https://github.com/raspberrypi/firmware/blob/master/boot/overlays/README").

The power figures quoted are the bare minimum to run the SoC with no peripherals. Often a 1A (2A for rpi4) is needed, especially if adding USB peripherals like keyboard, mouse, or wifi.

If unsure, use the official Raspberry Pi 15W USB-C power supply (5.1V / 3.0A DC). Similar CanaKit or mobile chargers should suffice. Using cheap power supplies off eBay is not recommended, as they often provide uneven and noisy current leading to unreliable operation.

It is possible to back-power some models from the USB ports, however this is not recommended, as the power in from the USB ports is not protected against surges whereas the main power supply is protected.

It is also possible to supply the required power via the GPIO pins.

Raspberry Pi 4 specific: The early rpi4 model has an issue with some USB-C cables, this is fixed in rev 1.2 and newer. Thermal throttling was also improvied, thus it is best practice to update the EEPROM as explained in the EEPROM section.

Power monitoring on a Raspberry PI 5 can be achieved by installing the following packages:

- collectd
- collectd-mod-exec
- collectd-mod-rrdtool
- collectd-mod-sensors

The following script can be used to provide rrd data for visualization:

```js
#!/bin/sh

HOSTNAME=$(uci get system.@system[0].hostname)

INTERVALTIME=10

while true; do
    TOTAL=$( /usr/bin/vcgencmd pmic_read_adc | awk '
    function clean(v){ gsub(/[AV]/,"",v); return v+0 }
    { split($0,a,"="); if ($1 ~ /_V/) V[$1]=clean(a[2]); if ($1 ~ /_A/) I[$1]=clean(a[2]) }
    END {
        total = 0
        total += V["3V7_WL_SW_V"] * I["3V7_WL_SW_A"]
        total += V["3V3_SYS_V"]   * I["3V3_SYS_A"]
        total += V["1V8_SYS_V"]   * I["1V8_SYS_A"]
        total += V["DDR_VDD2_V"]  * I["DDR_VDD2_A"]
        total += V["DDR_VDDQ_V"]  * I["DDR_VDDQ_A"]
        total += V["1V1_SYS_V"]   * I["1V1_SYS_A"]
        total += V["0V8_SW_V"]    * I["0V8_SW_A"]
        total += V["VDD_CORE_V"]  * I["VDD_CORE_A"]
        total += V["3V3_DAC_V"]   * I["3V3_DAC_A"]
        total += V["3V3_ADC_V"]   * I["3V3_ADC_A"]
        total += V["0V8_AON_V"]   * I["0V8_AON_A"]
        total += V["HDMI_V"]      * I["HDMI_A"]

        # Factor created by AI.
        total = total*1.145 + 0.588

        printf "%.3f\n", total
    }' )

    echo "PUTVAL \"$HOSTNAME/sensors-power/power\" interval=$INTERVALTIME N:$TOTAL"

    sleep $INTERVALTIME
done
```

Save the script (/usr/bin/rpi5-power.sh) and make it executable.

```js
chmod +x /usr/bin/rpi5-power.sh
```

You also need to make /dev/vcio (used by /usr/bin/vcgencmd to read the actual measurement data) accessable for the user nobody which is used by collectd. To do so persistently add the following lines to /etc/rc.local:

```js
mknod /dev/vcio c 100 0
chmod 666 /dev/vcio
```

After a reboot the statistics tab should be available in LuCI and there you can add the script. “Statistics” → “Setup” → “General Plugins” → “Exec” → “Add command for reading values”:

- Script: /usr/bin/rpi5-power.sh
- User: nobody
- Group: nogroup

Select “Save & Apply”. The power data should now be available: “Statistics” → “Graphs” → “Host”: YourHostname (default: OpenWrt) → “Sensors Tab”.
